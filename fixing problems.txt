SECTION 0: PROJECT CONTEXT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Goal: Obstacle avoidance + braking for a blind-assistance smart walker.
Safety requirement: HIGH accuracy — the user is blind and depends on it.

Hardware:
  - OAK-D camera (stereo depth) → mounted at ~70cm height
  - RPLIDAR C1 (360° 2D LiDAR)  → mounted at ~30cm height
  - Both on same axis, same forward direction (0° = forward)
  - Arduino Mega → controls motors, receives CMD: via Serial
  - Raspberry Pi → runs main.py, sends commands to Arduino

Project path: oak-template/
Key files:
  - main.py                         (pipeline + main loop)
  - utils/config.py                 (WalkerConfig dataclass — all params)
  - utils/fusion_layer.py           (FusionLayer — merges OAK + LiDAR)
  - utils/decision_engine.py        (DecisionEngine — picks command)
  - utils/lidar_analyzer.py         (LidarAnalyzer — background thread)
  - utils/corridor_analyzer.py      (CorridorAnalyzer — 7-zone depth)
  - utils/command_publisher.py      (CommandPublisher — rate limits sends)
  - utils/arduino_serial.py         (ArduinoSerial — bidirectional serial)
  - utils/tts_player.py             (TTSPlayer — voice feedback)
  - .env file → does NOT exist yet, needs to be created

Architecture flow:
  depth_frame → CorridorAnalyzer.analyze() → AnalysisResult
  LidarAnalyzer (background thread) → LidarScan (latest_scan)
  AnalysisResult + LidarScan → FusionLayer.fuse() → FusedAnalysis
  FusedAnalysis → DecisionEngine.decide() → DecisionResult
  DecisionResult → CommandPublisher.publish() → Arduino CMD

Zones (left to right): LEFT, L2, L1, CENTER, R1, R2, RIGHT
LiDAR arcs: front = ±30°, left = 30°→90°, right = -90°→-30°

BUG #1 [CRITICAL] — front_clear_mm NEVER passed to decide()
────────────────────────────────────────────────────────────
FusedAnalysis has field:  front_clear_mm: float
But in main.py lines 562-579, only these are forwarded:
  lidar_left_mm, lidar_right_mm, side_escape_left, side_escape_right

"front_clear_mm" is NEVER sent to DecisionEngine.decide().

Consequence: the _escape_latch_side in decision_engine.py can only
be released when fusion_reason changes AWAY from "lidar_veto_side_escape".
But fusion_reason stays "lidar_veto_side_escape" as long as LiDAR
front_min_mm < LIDAR_SAFETY_MM (600mm). Since the latch release has
no direct access to front_min_mm, it can NEVER proactively release —
it relies on a side-effect of fusion changing state.

Proof from log (output_scenario.txt lines ~282-483):
  stable_count=4 → GO:L1, then stable_count=153 → still GO:L1
  L=2300+mm (left very open), R=150mm (right blocked), front=450mm
  System stays stuck in GO:L1 for ~150 frames (~5 seconds)
  Only escapes when _is_free_candidate accidentally triggers

BUG #2 [HIGH] — SIDE_PREFER_MARGIN=0.15 causes flip-flop
───────────────────────────────────────────────────────────
In config.py line 134: SIDE_PREFER_MARGIN: float = 0.15
In from_env() line 249: default "0.15"

From log (output_scenario.txt lines 192-224), during FREE mode:
  [Decision] Side zone LEFT preferred:  safety=1.20 vs CENTER=1.07  diff=0.13
  [Decision] Side zone RIGHT preferred: safety=1.25 vs CENTER=1.07  diff=0.18
  [Decision] Side zone LEFT preferred:  safety=1.26 vs CENTER=1.08  diff=0.18
  (flips every few frames between LEFT and RIGHT)

The difference is 0.13-0.18 — exactly straddling the 0.15 threshold.
Each frame noise shifts it above/below → random left/right oscillation.
This means the system has NO stable direction when path is open.

BUG #3 [HIGH] — Height difference causes false LiDAR vetoes
───────────────────────────────────────────────────────────
LiDAR at 30cm reads chair legs / table bases as obstacles.
Camera at 70cm sees over them → says path is clear.
Result: fusion triggers "lidar_veto_side_escape" for non-obstacles,
forcing unnecessary GO:L1/R1 commands when path is actually free.

From log: lidar_front=450mm while depth_mm=847mm (camera sees clear).
The camera p20 in front zones was ~800-1478mm during this sequence —
high enough to be "clear" by camera standards but LiDAR keeps vetoing.

───────────────────────────────────────────────────────────
BUG #4 [MEDIUM] — Side arc too wide (30°-90°) measures walls
───────────────────────────────────────────────────────────
In config.py lines 166-167:
  LIDAR_SIDE_ARC_START_DEG: float = 30.0
  LIDAR_SIDE_ARC_END_DEG: float   = 90.0

An arc of 30°-90° includes 90° (pure sideways) → measures wall distance.
For a walker that needs to turn slightly, we need forward-lateral info
(~45°-75°), not pure lateral. This inflates "side_escape" readings and
makes the system think there's more room than actually exists forward-left.

───────────────────────────────────────────────────────────
BUG #5 [MEDIUM] — Escape latch fragile: clears on single frame
───────────────────────────────────────────────────────────
In decision_engine.py lines 318-320:
  else:
      # Not in side-escape mode: clear the latch
      self._escape_latch_side = ""

If fusion_reason is NOT "lidar_veto_side_escape" for even ONE frame
(e.g. a noisy scan briefly shows front_clear=True), the latch clears.
Next frame it re-engages and may pick a DIFFERENT direction.
This creates: go-left, stop, go-right, stop pattern.

───────────────────────────────────────────────────────────
There is no .env file in oak-template/. All tunable parameters
are using hardcoded defaults from config.py. This makes it hard
to tune without code changes. A .env file needs to be created.

   Claude proposed checking side distances to release front latch.
   This is logically incorrect. Use LIDAR_SAFETY_MM for front check.

   The log shows differences of 0.13-0.17 causing flip-flop.
   0.20 fixes this without being so aggressive it prevents needed turns.
   0.28 would suppress too many legitimate steering corrections.

❌ DO NOT train the camera on scenarios.
   The camera sees obstacles correctly. The bug is in decision logic,
   not in camera perception. Training is expensive and unnecessary.

❌ DO NOT reorganize camera=direction / LiDAR=stopping architecture.
   The hybrid fusion is architecturally correct. Fix the bugs, not the design.

```python
    # Only trust LiDAR front veto when:
    #   a) LiDAR front is very close (< LIDAR_FRONT_TRUST_MM), AND
    #   b) Camera also sees something close (oak_front_mm < LIDAR_CAMERA_AGREE_MM)
    # This prevents chair legs / low objects from triggering unnecessary stops.
    oak_front = self._oak_front_mm(oak_analysis)

    camera_agrees = oak_front < self._cfg.LIDAR_CAMERA_AGREE_MM

    if not lidar_very_close:
        # LiDAR sees obstacle but it's not very close — likely low object.
        # Ignore veto, treat as clear with slight confidence boost.
        return FusedAnalysis(
            oak_analysis=oak_analysis,
            has_emergency=False,

            side_escape_left=bool(scan.side_escape_left),
            side_escape_right=bool(scan.side_escape_right),

            lidar_left_mm=float(scan.left_min_mm),
            lidar_right_mm=float(scan.right_min_mm),
        )

        # LiDAR sees real close obstacle with side escape available.
        # Only activate side-escape if camera somewhat agrees.
        if camera_agrees:
            return FusedAnalysis(
                oak_analysis=oak_analysis,
                has_emergency=False,
                front_clear_mm=float(lidar_front_mm),

                side_escape_right=bool(scan.side_escape_right),

                front_clear_mm=float(oak_front),
                side_escape_left=bool(scan.side_escape_left),
                side_escape_right=bool(scan.side_escape_right),
                lidar_active=True,
                confidence_boost=+0.03,
                fusion_reason="lidar_low_obstacle_camera_clear",
                lidar_left_mm=float(scan.left_min_mm),
                lidar_right_mm=float(scan.right_min_mm),
            )

    # No side escape — true emergency only if camera also agrees.
    if camera_agrees:
        fused_oak = self._override_emergency(oak_analysis, True)
        return FusedAnalysis(
            oak_analysis=fused_oak,
            has_emergency=True,
            front_clear_mm=float(lidar_front_mm),
            side_escape_left=False,
            side_escape_right=False,
            lidar_active=True,
            confidence_boost=+0.05,
            fusion_reason="lidar_veto_emergency",
            lidar_left_mm=float(scan.left_min_mm),
            lidar_right_mm=float(scan.right_min_mm),
        )
    else:
        # LiDAR close but camera clear → low obstacle, ignore.
        return FusedAnalysis(
            oak_analysis=oak_analysis,
            has_emergency=False,
            front_clear_mm=float(oak_front),
            side_escape_left=False,
            side_escape_right=False,
            lidar_active=True,
            confidence_boost=+0.03,
            fusion_reason="lidar_low_obstacle_camera_clear",
            lidar_left_mm=float(scan.left_min_mm),
            lidar_right_mm=float(scan.right_min_mm),
        )
```

ولأن `FusionLayer` يحتاج الـ config الآن، عدّل الـ `__init__`:
```python
def __init__(self, lidar: LidarAnalyzer, cfg):
    self._lidar = lidar
    self._cfg = cfg
```

وفي `main.py` حيث تُنشئ `FusionLayer`:
```python
fusion = FusionLayer(lidar, cfg)  # أضف cfg
```

### التعديل 3: `utils_decision_engine.py` — فك الـ latch تلقائياً

في دالة `decide()` أضف مباشرة بعد سطر `self._update_ema(analysis)`:

```python
# ── Auto-release escape latch when front is clear on both sensors ──
if self._escape_latch_side:
    # Check camera front clearance
    cam_front_p20 = min(
        (analysis.corridors[z].p20_depth
         for z in ("L1", "CENTER", "R1")
         if analysis.corridors.get(z)
         and analysis.corridors[z].p20_depth > 0
         and analysis.corridors[z].valid_ratio >= cfg.MIN_VALID_RATIO),
        default=0.0,
    )
    lidar_front_released = (
        lidar_left_mm > cfg.ESCAPE_RELEASE_FRONT_MM
        and lidar_right_mm > cfg.ESCAPE_RELEASE_FRONT_MM
    )
    camera_front_released = cam_front_p20 > cfg.ESCAPE_RELEASE_CAMERA_MM

    # Release if BOTH sensors agree front is clear
    if lidar_front_released and camera_front_released:
        print(
            f"[Decision] Escape latch RELEASED: "
            f"cam_p20={cam_front_p20:.0f}mm "
            f"lidar_L={lidar_left_mm:.0f}mm lidar_R={lidar_right_mm:.0f}mm"
        )
        self._escape_latch_side = ""
```

## ملخص ما يتغير

| المشكلة | السبب | الحل |
|---------|-------|------|
| LiDAR يرى عوائق وهمية | ارتفاعه منخفض يرى أرجل الكراسي | veto يحتاج موافقة الكاميرا |

| قرارات يمين/يسار تتقلب | SIDE_PREFER_MARGIN على حافة النيف | رفعه من 0.18 إلى 0.28 |
